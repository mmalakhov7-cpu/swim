# CLAUDE.md

Контекст и правила работы для ИИ-агента над этим проектом.
**Детальные требования — в `ТЗ.md`. Этот файл — про то, КАК работать.**

---

## Что это за проект

Веб-приложение (обычный HTML/CSS/JS) — **дневник тренировок в бассейне**.
Пловец во время заплыва жмёт кнопки-отсечки, приложение считает время и дистанцию,
сохраняет результаты локально, строит графики прогресса и копит общий объём.

Полное описание функциональности, экранов, модели данных и формул — в **`ТЗ.md`**.
Перед любой реализацией прочитай `ТЗ.md` целиком, особенно **раздел 6** (логика
таймеров и отрезков) — там больше всего тонкостей.

---

## Технологический стек и ограничения

- **Vanilla HTML + CSS + JavaScript (ES-модули). Без фреймворков и без сборки.**
  Приложение — статический сайт, запускается локальным сервером.
- **Chart.js — только локальной копией** в `vendor/`. Никаких внешних CDN.
- Хранение — **`localStorage`**, доступ ТОЛЬКО через модуль `src/storage.js`
  (чтобы позже без переделок заменить на IndexedDB). Не обращаться к `localStorage`
  напрямую из других модулей.
- **Полный офлайн:** в рантайме не должно быть ни одного сетевого запроса.
- Целевая платформа №1 — **телефон в портретной ориентации** (у бортика бассейна).

---

## Приоритеты качества (в порядке важности)

1. **Корректность логики** таймеров, отсечек, дистанции/бассейнов, накопления объёма
   (см. `ТЗ.md`, разделы 6–7). Это ядро — здесь ошибки недопустимы.
2. **Удобство у бассейна:** крупные тач-таргеты, высокий контраст, читаемость на солнце,
   Undo последней отсечки, экран не гаснет во время тренировки.
3. **Надёжность данных:** ничего не теряется, есть экспорт/бэкап в JSON.
4. Чистый, читаемый, модульный код.

---

## Соглашения по коду

- Один модуль — одна ответственность (см. предлагаемую структуру в `ТЗ.md`, раздел 10).
- Время храним и считаем в **миллисекундах**; расстояния — в **метрах**. Форматирование —
  только на выводе (модуль `timer.js`).
- Производные величины (бассейны, темп, остаток) **вычислять**, а не дублировать в хранилище.
- Именование сущностей — как в модели данных `ТЗ.md` (раздел 5): `WorkoutSession`,
  `TaskResult`, `Split`, `WorkoutTemplate`, `TaskTemplate`, `Settings`.
- Комментарии и UI-тексты — на русском (пользователь русскоязычный). Имена
  переменных/функций — на английском.
- Никаких «магических чисел»: длина чаши, шаги отсечки (25/50), целевые дистанции —
  из данных/настроек, а не зашиты в логику.

---

## Важные архитектурные решения (не переигрывать без спроса)

- **Кнопки отсечки** — две отдельные: `+25 м` и `+50 м`. Каждое нажатие пишет `Split`
  и продвигает дистанцию/бассейны. Обязателен **Undo**.
- **Отрезки 25/50/100 м** считаются по маркерам ТОЧНО как в `ТЗ.md` §6.3. Если точной
  границы нет — показывать `«—»`. **Не интерполировать и не выдумывать значения.**
- **Длина чаши = 25 м** по умолчанию (настраиваемо). Из этого: 400 м = 16 бассейнов,
  2000 м = 80.
- **Автогенерация последовательности заданий** — правила будут позже. Сейчас: сделать
  ручные шаблоны и **модуль-заглушку** `src/workoutGenerator.js` с чётким интерфейсом,
  чтобы правила добавились без переделки моделей.

## Как работать над задачей

1. Прочитать `ТЗ.md` целиком.
2. Собрать **скелет проекта** по структуре из `ТЗ.md` §10 (файлы + пустые модули).
3. Реализовать сначала **ядро активной тренировки** (§4.3 и §6) — это главная ценность,
   и на нём проще всего ошибиться. Проверить сценарий: старт → серия `+25`/`+50` → Undo
   → `→` следующее задание → завершение → сохранение.
4. Затем **хранение и дашборд** (§4.1, §5, §7), затем **аналитику/графики** (§4.4).
5. Затем нефункциональные вещи: офлайн/PWA, Wake Lock, экспорт JSON (§8).
6. Прогонять по **чек-листу приёмки** `ТЗ.md` §11 — это Definition of Done.

При неоднозначности: сначала смотри «Решения по умолчанию» (`ТЗ.md` §12). Если там нет —
выбери разумный вариант, **явно отметь допущение** и продолжай, не блокируясь.

## Что НЕ делать

- Не тянуть внешние CDN/зависимости в рантайм.
- Не обращаться к `localStorage` мимо `storage.js`.
- Не интерполировать времена отрезков (см. §6.3).
- Не хардкодить длину чаши и дистанции в логике.
- Не отправлять данные пользователя куда-либо по сети.
